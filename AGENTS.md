## 保護対象ファイル

- 保護対象ファイルの変更は、人間の明示指示があった場合に限り可能。
- その際のコミットメッセージは `docs(approved):` を先頭に付けること。
- このプレフィックスのない `docs/`・`AGENTS.md` 変更は運用違反として扱う。

## GitHub / Issue運用

### 原則

- 全ての実装作業はIssueに基づいて行う。Issueのない実装を開始しない。
- 1 Issue = 1つの限定されたスコープ。Issue本文の「スコープ」「受け入れ条件」の範囲外は変更しない。
- Issueの作成・優先順位・クローズ判断は人間が行う。CodexはIssueを作成しない。作業中に発見した問題は、PRの報告欄に「Issue候補」として記載するに留める。

### ブランチ / コミット / PR

- ブランチ名: `issue-<番号>-<短い英語スラッグ>`（例: issue-3-game-core-tick）
- コミットメッセージ先頭に `#<Issue番号>` を付ける
- 作業完了時はPRを作成し、本文に `Closes #<番号>` と標準報告形式（変更ファイル / 実行コマンドと結果 / 実装した仕様節番号 / 迷った仕様節番号）を書く
- 1 PR = 1 Issue。複数Issueをまとめたブランチ・PRを作らない
- type:balance のPRに UI変更・機能追加を含めない（既存の混在禁止規則と同じ）

### レビューフロー

1. Codex: Issue着手 → 実装 → 標準確認コマンド実行 → PR作成
2. 人間: PRの報告とsimログをClaude（チャット）のレビューにかける
3. 人間: 採用ならマージ、差し戻しならPRにコメントで修正指示
- Codexはセルフマージしないこと
