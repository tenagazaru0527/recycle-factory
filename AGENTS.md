## 運用規約バージョン

- 本規約は v3（速度優先）。チャットClaude のPR単位レビューを外し、Issue 起草・
  docs/design.md 改訂・バランス掃引を実装AIへ移譲して人間との往復を減らす。
- 速度優先でも「維持する規定」（末尾）は緩和しない。特に Stop Condition は全項目を維持する。

## 保護対象ファイル

- 保護対象は `docs/`・`AGENTS.md`・`CLAUDE.md`・`.github/workflows/`・`package.json`。
- 保護対象の変更は PR 経由で行い、人間のマージを承認とみなす。
- `docs/design.md` は実装AIが PR 内で改訂できる（人間のマージが承認を意味する）。
- `AGENTS.md`・`CLAUDE.md`・`.github/workflows/` の変更は、人間の明示指示がある場合に限る。
- `docs(approved):` プレフィックスは、人間が main へ直接コミットする場合のみ付ける。
  PR 経由の変更には不要。
- CI の protected-files ジョブは、PR が変更する保護対象ファイルをジョブサマリに表示する
  非ブロッキング通知であり、マージをブロックしない。人間はマージ前にこれを確認する。

## GitHub / Issue運用

### 原則

- 全ての実装作業はIssueに基づいて行う。Issueのない実装を開始しない。
- Issueの作成・本文修正・クローズは実装AIが行う。実装AIが docs/design.md とコードを
  読んでIssue本文を起草する。人間はタイトルと目的を1行で指示し、何に着手するか
  （優先順位）を決める。
- Issueの粒度: マイルストーン単位・複数受け入れ条件を許容する。1行変更のような小変更を
  単独Issue化しない。着手後は Issue本文の「スコープ」「受け入れ条件」の範囲を守る。
- 作業中に発見した範囲外の問題は、新規Issueとして起票するか、PRの報告欄に
  「Issue候補」として記載する。

### ブランチ / コミット / PR

- ブランチ名: `issue-<番号>-<短い英語スラッグ>`（例: issue-3-game-core-tick）
- コミットメッセージ先頭に `#<Issue番号>` を付ける
- 作業完了時はPRを作成し、本文に `Closes #<番号>` と標準報告形式（変更ファイル / 実行コマンドと結果 / 実装した仕様節番号 / 迷った仕様節番号）を書く
- 1 PR = 1 Issue。複数Issueをまとめたブランチ・PRを作らない
- バランス数値変更と機能・UI変更を同一PRに混在させない。type:balance のPRに UI変更・機能追加を含めない

### 並行実行 / worktree

- worktree を用いた2タスク同時進行を標準とする。
- worktree はリポジトリの親ディレクトリ配下（例: ../recycle-factory-<用途>）に作成し、
  /tmp 配下には作らない。
- 完了時に `git worktree remove` する。

### バランス調整

- パラメータ範囲と目的関数（G-1〜G-5）を与えられた場合、実装AIが掃引・選定・適用・
  再測定まで一括で行う。人間は結果の採否のみ判断する。
- バランス数値の変更は type:balance のPRとし、機能・UI変更と混在させない
  （「ブランチ / コミット / PR」の混在禁止を参照）。

### レビューフロー

1. 実装AI: Issue起草 → 着手 → 実装 → 標準確認コマンド実行 → PR作成。セルフマージはしない。
2. Codexレビュー: 次の2種のPRでのみ必須。それ以外（docs改訂・マイルストーン完了処理など）は不要。
   - game-core.js の変更を含むPR
   - type:balance のPR
3. チャットClaude: PR単位のレビューは行わない。関与は設計判断・測定結果の解釈・
   マイルストーンの締めに限る。
4. 人間: マージ判断。マージは人間のみが行う。

## 維持する規定（速度優先でも緩和しない）

- Stop Condition は全項目を維持する（速度優先でも緩和しない）。次のいずれかを検出したら、
  実装を止めて具体的な判断材料を報告する。
  - 指示・Issue本文・AGENTS.md・docs/design.md の間に矛盾を検出した
    （解消可能かどうかを問わず、実装前に報告する）
  - docs/design.md に未定義の仕様が実装に必要になった
  - Issueスコープ外・許可範囲外のファイル変更が必要になった
  - main やブランチの状態が想定と異なる（退行・基点ズレ・履歴の欠落を検出した）
  - 修正ループが3往復で収束しない / 同一原因のCI失敗が3回続いた
  - データ消失につながる操作が必要になった
- CI の test ジョブ（`npm test`）はブロッキングのまま維持する。
- docs/design.md を仕様の正本とする原則。
- バランス数値変更と機能・UI変更を同一PRに混在させない。
- main への直接push・force push の禁止（人間が保護対象を main へ直接コミットする場合を除く）。
- マージは人間のみが行う。
